# TAQWIN MASTER RECOVERY — PART 1: FAILURE AUDIT
# Date: 2026-04-14 | Status: ACTIVE EXECUTION | Author: TAQWIN

---

## SYSTEM CONTEXT

This system is NOT early-stage. It is a mid-to-advanced AI agent interface with:
- Chat pipeline with streaming
- MCP integration (disabled by flag)
- Output interpreter layer (P2.6)
- Tool approval modal
- Response time tracking

BUT connections between layers are broken or misaligned. This document maps
every confirmed failure zone to exact file and line evidence, then specifies
the exact restoration required.

---

## TARGET ARCHITECTURE

```
User → ChatPane.tsx
     → ChatService.ts         (state + queue + lifecycle)
     → OutputInterpreter.ts   (classify BEFORE any UI touch)
     → KnezClient.ts          (SSE streaming from backend)
     → KNEZ FastAPI backend   (completions.py → LocalBackend)
     → Ollama (model)

Tool path (MCP_ENABLED=true only):
ChatService → runPromptToolLoop → McpOrchestrator → TAQWIN_V1
           ← tool result ← model re-ask ← interpreter validated
```

## SYSTEM LAWS (NON-NEGOTIABLE)

1. UI must NEVER render raw model output
2. Model output must ALWAYS be interpreted before use
3. Tool calls must NEVER be visible to user
4. System payloads must NEVER reach UI
5. Final response MUST ALWAYS be generated by AI
6. Streaming must only emit VERIFIED HUMAN TEXT
7. No regex-based sanitization anywhere in pipeline

## FAILURE ZONE AUDIT

### F1 — CRITICAL: LocalBackend Non-Streaming
File: KNEZ/knez/knez_core/models/local_backend.py, line 31
```python
payload = {"model": self.model_id, "prompt": prompt, "stream": False}
```
Lines 87-90:
```python
await stream_callback(text)   # entire response as one callback
yield text                     # entire response as one yield
await stream_callback("__DONE__")
yield "__DONE__"
```
IMPACT: SSE emits ONE delta.content event with full response.
User sees blank for 3-10s then entire message appears at once.
KnezClient.ts line 1082 logs "STREAM CHUNK: <entire response>" — confirms this.

FIX: Convert _fetch to _stream_tokens using Ollama stream=True API.
Each token {"response":"tok","done":false} must yield individually.

---

### F2 — CRITICAL: Recovery Response Bypasses Interpreter
File: ChatService.ts, lines 1251-1257
```typescript
for await (const chunk of knezClient.chatCompletionsStream(...)) {
  this.appendToMessage(assistantId, chunk);  // DIRECT TO UI — LAW #1,#2,#6 VIOLATED
  recoveryText += chunk;
}
```
IMPACT: If recovery prompt returns JSON/tool_call, it goes straight to UI.
The entire interpreter layer exists to prevent this. Recovery bypasses it.

FIX: Buffer recovery stream, classify via interpretOutput, only stream
plain_text to UI. Hard fallback if recovery also non-plain.

---

### F3 — HIGH: sanitizeOutput Uses Regex (Law #7 Violated)
File: ChatService.ts, lines 1170-1173
```typescript
const stripped = text
  .replace(/\{.*?"tool_call".*?\}/gs, "")   // REGEX — LAW #7
  .replace(/\{.*?"tool_calls".*?\}/gs, "")  // REGEX — LAW #7
```
REDUNDANT: Called only on interpreter-validated plain_text. Plain text
cannot contain complete tool JSON (interpreter fast-path: !startsWith("{")).
FIX: Replace body with return text.trim()

### F4 — HIGH: extractToolCall Dead Code + Regex

const match = text.match(/\{[\s\S]*\}/);  // REGEX — LAW #7

Lint warning at line 1177: "declared but value is never read"
Superseded entirely by interpretOutput() from OutputInterpreter.ts.
FIX: Remove method.

### F5 — MEDIUM: runNativeToolLoop maxSteps = 8
File: ChatService.ts, line 772
runPromptToolLoop was fixed to 3. runNativeToolLoop is still 8.
Both are dead code but must be consistent.
FIX: const maxSteps = 3

### F6 — MEDIUM: interpretBuffer Bloat After Classification
File: ChatService.ts, lines 1390-1406
After plain_text classification, interpretBuffer still grows with every chunk
via "interpretBuffer += chunk" at line 1394. The buffer is never read again
after classification. For a 5000-token response, entire response duplicated in RAM.
FIX: Only accumulate interpretBuffer when outputClass === null.

---

### F7 — MEDIUM: No Finally Block for Streaming State Reset
File: ChatService.ts, deliverQueueItem try/catch
streaming=false reset exists in success path and some catch paths.
Edge-case errors can leave streaming=true, disabling send button permanently.
FIX: Add finally block guarded by streamId identity check.

---

### F8 — LOW: Agent Runtime Not Connected
App.tsx imports AgentPane. Agent loop (Understand→Plan→Execute→Observe→Respond)
not wired into chat pipeline. DEFERRED to separate agent integration phase.

## ROOT CAUSE SUMMARY

1. LocalBackend was written for correctness, not streaming UX
2. generateRecoveryResponse was written before interpreter existed, never updated
3. sanitizeOutput and extractToolCall predate P2.6, not removed when superseded
4. Agent runtime wiring never completed from feature branch
