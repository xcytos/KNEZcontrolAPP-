# CP5: Full Feature Surface Realization Plan

Based on the Gap Analysis of `KNEZ/knez` capabilities vs `Control App` UI, here is the execution plan for Checkpoint 5.

## 🎫 Ticket Set #1 (10 Mandatory Tickets)

### 1. CP5-1: Full Session Lineage & Resume Inspector
*   **Objective**: Visualize conversation branches and allow resuming from past snapshots.
*   **Backend**: `GET /sessions/{id}/resume_snapshot`, `POST /sessions/{id}/resume`
*   **UI**: "Lineage" tab in Chat Sidebar showing a tree of forks. Clicking a node shows snapshot details and "Resume Here" button.

### 2. CP5-2: Memory Evidence Drill-down
*   **Objective**: Inspect the "why" behind memory items.
*   **Backend**: `GET /memory/{id}`
*   **UI**: Clickable Memory items in Explorer -> Modal showing `confidence` score, `retention_policy`, and linked `evidence_event_ids`.

### 3. CP5-3: Event Severity & Source Visualization
*   **Objective**: High-signal observability for system events.
*   **Backend**: `GET /events` (enhanced parsing)
*   **UI**: `KnezEventsPanel` update: Color-coded rows (Red=Error, Yellow=Warn), Filter dropdowns (Source, Severity).

### 4. CP5-4: Influence & Challenge Lifecycle Panel
*   **Objective**: Granular control over behavioral contracts.
*   **Backend**: `GET /operator/influence/contracts`, `POST /operator/influence/domain`
*   **UI**: `GovernancePanel` expansion: List active `InfluenceContract`s with their `max_weight` and `scope`. Toggle switches for specific domains.

### 5. CP5-5: Replay Timeline with Phase Segmentation
*   **Objective**: Post-hoc analysis of session execution.
*   **Backend**: `GET /sessions/{id}/replay`
*   **UI**: New `ReplayPane`: Horizontal timeline visualization showing phases (`start_time` to `end_time`) and event density.

### 6. CP5-6: MCP Health & Capability Explorer
*   **Objective**: Deep dive into tool availability.
*   **Backend**: `GET /mcp/registry`
*   **UI**: `McpRegistryView` update: Show `capabilities` list as tags. Visual "Health" indicator based on `status`.

### 7. CP5-7: Detailed Cognitive State View
*   **Objective**: Expose hidden subsystem states.
*   **Backend**: `/state/governance`, `/state/influence`, `/state/taqwin`
*   **UI**: `CognitivePanel` tabs: "Overview" (existing), "Governance State", "Taqwin Context".

### 8. CP5-8: Audit & Consistency Validation Panel
*   **Objective**: Verify system integrity.
*   **Backend**: `GET /audit/consistency`
*   **UI**: New section in `InfrastructurePanel`: "System Integrity" showing pass/fail checks from the audit endpoint.

### 9. CP5-9: Orchestration Fast-path Robustification
*   **Objective**: Ensure "Launch" is idempotent and verifiable.
*   **Logic**: Update `useSystemOrchestrator`: If `/health` is OK, skip shell command entirely. Force "Running" state only after positive health check.

### 10. CP5-10: Unified Status Reconciliation Engine
*   **Objective**: Single source of truth for UI state.
*   **Logic**: Global React Context (`StatusProvider`) that polls `/health` and `/state/overview` every 2s. All other components (Chat, Memory, etc.) read "Connected" state from here, not local state.

## 🔄 Execution Loop (Phase 3-5)
1.  **Implement** each ticket sequentially.
2.  **Verify** via manual signal checks (logs/UI feedback).
3.  **Serialize** results into `.taqwin` records.
4.  **Commit** final state.
